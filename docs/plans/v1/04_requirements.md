## 0. 개요

### 0-1. 목적·범위

본 문서는 `00_meeting_brief.md`(역설계 회의 3라운드 종합)에서 **확정된 제품 정의·결정 로그 11건·로드맵 4레이어·도메인 모델·IPC 통합 계약**을 기능 요구사항(FR)·비기능 요구사항(NFR)·제약으로 정형화한다. 요구사항을 발명하지 않으며, 브리프의 결정을 뒤집지 않는다.

- **대상 제품**: Control Tower — 로컬에서 여러 Claude Code/PowerShell 세션을 앱이 직접 소유(ConPTY 임베드)하여, 의도별 세션 프로파일로 띄우고·IPC 채널로 협업시키며·토큰·프롬프트 자산까지 관제하는 **단일 파워유저용 AI 세션 관제탑**.
- **1차 정체성**: AI 세션 오케스트레이션(결정 로그 #1). 터미널 임베드(ConPTY)는 그 가치를 실현하는 substrate(결정 로그 #4 — ConPTY 0순위 기반).
- **v1 In-Scope**: 로컬 전체 — ConPTY 임베드·세션 소유/제어·세션 프로파일·IPC 오케스트레이션(로컬 파일 기반)·토큰 관측·프롬프트 자산 편집.
- **v1 Out-of-Scope**: ⑤ 멀티디바이스 원격제어(HTTP relay 기반) = **v2**(결정 로그 #2). 프롬프트 자산의 검증·스키마화(파일 읽기/쓰기만 v1). pane 분할은 후속 마일스톤(결정 로그 #8).
- **경계(다른 문서와의 관계)**: 사용자 유형·페르소나=03 / 기능 분해(FN)=05 / 화면=07 / DB·프로파일 영속화 형태=09 / 터미널 엔진·아키텍처=10. 본 문서는 그 상위의 "무엇을·왜"만 정의한다.

### 0-2. ID·카테고리 체계

- **FR-###**: 기능 요구사항, 3자리 zero-pad. 본 문서가 발번·freeze한다.
- **NFR-###**: 비기능 요구사항, 3자리 zero-pad. 본 문서가 발번·freeze한다.
- **8 공통 카테고리(3자 대문자)**: 본 문서가 도메인(브리프의 5역량+도메인 모델)에서 파생해 **1회 확정(FREEZE)**한다. FR과 하위 기능정의(FN, 05)가 **동일 코드셋을 공유**하여 FR↔FN 추적의 기반이 된다. 하위 agent(05·07·09)는 새 카테고리를 만들지 않고 참조만 한다. 각 코드는 영어 단어의 약칭이며 **원어** 열에 출처를 밝힌다.

### 0-3. 우선순위 정의 (MoSCoW)

| 등급 | 정의 | 판정 기준 |
|---|---|---|
| **Must** | v1이 가치를 내려면 반드시 있어야 함. 없으면 제품 정체성 붕괴 | 오케스트레이션 substrate·핵심 5역량의 필수 경로 |
| **Should** | 중요하나 없어도 v1 출시 가능. 초기 마일스톤 직후 우선 보강 | 품질·편의·관측 심화 |
| **Could** | 있으면 좋음. 여력/후속 마일스톤에서 | pane 분할·상태 복원 등 브리프가 "후속"으로 명시한 항목 |
| **Won't (v2)** | v1 범위 밖. 참고로만 기록 | 멀티디바이스 원격제어(§4-2 Out-of-Scope) |

### 0-4. 표기 규칙

- **AC**(수용 기준, Acceptance Criteria): 각 FR이 "충족되었다"를 판정하는 관찰 가능 조건. `/`로 복수 조건 구분.
- **근거**: 브리프 결정 로그(#n)·로드맵 레이어·도메인 모델을 출처로 표기(정형화 추적성).
- **관련 숙제**: §5 후속 숙제 ①~⑧ 번호. 구현 전 확정 필요 항목.
- **측정 지표**: NFR의 정량 목표. 값은 잠정(가정) — §5·확인질문에서 확정 대상 표기.
- **[L0~L3]**: 로드맵 레이어. **[v2]**: Out-of-Scope 참고 항목.

## 2. 기능 요구사항(FR) 상세

### 2-1. TRM — 터미널 임베드·렌더·탭 [L0 substrate]

#### [FR-001] 앱-소유 pwsh ConPTY 세션 생성(spawn) — Must
앱이 `CreatePseudoConsole` 기반으로 pwsh 프로세스를 직접 생성하고 그 PTY를 소유한다. 외부 프로세스 attach는 불가하므로(결정 로그 #7) 오직 앱이 spawn한 세션만 대상이 된다.
- AC: 신규 세션 요청 시 pwsh 프로세스가 ConPTY와 함께 생성됨 / 앱 종료 시 소유 세션이 정리됨 / 외부(앱 실행 전부터 떠 있던) 프로세스는 목록/제어 대상이 아님
- 근거: 결정 로그 #4(ConPTY 0순위), #7(앱-소유 세션만)
- 관련 숙제: —

#### [FR-002] ConPTY 입출력 파이프 바인딩 — Must
세션의 입력(주입 경로)과 출력(캡처 소스)을 앱이 확보한다. 입력 경로는 렌더 완성도와 무관하게 확보되어 커맨드/스킬 주입을 조기 가능하게 한다. 주입은 `TermPTY.WriteToTerm`, 출력 캡처는 `InterceptOutputToUITerminal`/`GetConsoleText`로 실현한다.
- AC: 세션 출력이 화면에 렌더되고 앱이 캡처 가능 / 앱이 임의 텍스트를 주입하면 세션 프롬프트에 반영됨 / 파이프 오류 시 세션 상태가 error로 전이
- 근거: 결정 로그 #4, 브리프 §2(주입 경로 ①단계 확보)
- 관련 숙제: —

#### [FR-003] VT 시퀀스 렌더링 (엔진 위임) — Must
ANSI/VT 이스케이프 시퀀스 파싱·렌더링을 자체 구현하지 않고 공식 Windows Terminal 렌더러(EasyWindowsTerminalControl)에 위임한다.
- AC: 색·커서 이동·소거 등 VT 시퀀스가 화면에 정확히 반영됨 / 엔진 접근이 경계 인터페이스(`ITerminalSession`) 1곳으로 격리되어 교체 가능
- 근거: 결정 로그 #6(공식 WT 렌더러 위임)
- 관련 숙제: —

## 3. 비기능 요구사항(NFR) 상세

> 측정 지표 값은 잠정 목표(가정)이며 §5 후속 숙제 ③(렌더 성능)·기술 스펙(10)에서 확정한다.

### 3-1. 성능 (Performance)

#### [NFR-001] 터미널 렌더 처리량 — Must
- 지표: 대량 출력(예: `cat` 대용량, 빌드 로그) 스트리밍 시 UI 프레임 유지 ≥ 30fps / 스크롤 리페인트 지연 ≤ 50ms
- 관련: FR-004·FR-005·FR-007 · 관련 숙제 ③
- 검증: 10만 라인 출력 스트레스에서 프레임 드랍/멈춤 없음(기술 스펙 PoC)

#### [NFR-002] 커맨드 주입 지연 — Must
- 지표: 커맨드 전송 요청→PTY write 반영 ≤ 100ms (로컬)
- 관련: FR-014·FR-027

#### [NFR-003] 다중 세션 동시성 — Should
- 지표: 8개 세션 동시 running 시 idle 앱 CPU ≤ 10%, 메모리 ≤ 800MB (기준 하드웨어)
- 관련: FR-017 · NFR-012

#### [NFR-004] IPC watch 반영 지연 — Should
- 지표: 채널 inbox.log 파일 변경→앱 UI 반영 ≤ 2s (파일 기반 watch, f_relay 폴링 기본 2s와 정합)
- 관련: FR-024·FR-028

#### [NFR-005] 토큰 파싱 성능 — Should
- 지표: jsonl 증분 파싱 — 신규 append분만 재파싱, 100MB 트랜스크립트에서 집계 갱신 ≤ 500ms
- 관련: FR-031

### 3-2. 보안·안전 (Security/Safety)

#### [NFR-006] 로컬 전용·외부 포트 미개방 — Must
- 지표: 앱이 여는 수신 소켓/포트 = 0 (relay 서버는 앱 외부, v2)
- 관련: FR-038 · 결정 로그 #2

#### [NFR-007] 파괴적 명령 실행 전 확인 — Should
- 지표: 정의된 위험 패턴 주입 시 100% 확인 게이트 경유
- 관련: FR-037 · 관련 숙제 ⑧

#### [NFR-008] 자산 편집 경로 안전 — Must
- 지표: 자산 편집 대상 경로가 `~/.claude` 밖으로 벗어나는 접근 0(경로 탈출 `..` 차단)
- 관련: FR-033~036

### 3-3. 가용성·신뢰성 (Availability/Reliability)

#### [NFR-009] 세션 크래시 격리 — Must
- 지표: 1개 세션/PTY 비정상 종료가 앱·타 세션에 전파되지 않음(격리 100%)
- 관련: FR-015·FR-016

#### [NFR-010] watcher/relay stale 복구 — Should
- 지표: stale `.watcher_<as>.pid` 감지 시 skill_ipc_control 가드와 정합되게 복구(자동 흡수), 사용자 동의 정책 준수
- 관련: FR-024 · skill_ipc_control Error Handling

#### [NFR-011] 상태 영속 원자성 — Must
- 지표: 프로파일/자산 저장 시 부분 기록으로 인한 손상 0 (원자적 write, temp→rename)
- 관련: FR-020·FR-035 · 관련 숙제 ①

### 3-4. 확장성 (Scalability)

#### [NFR-012] 동시 세션 수 확장 — Should
- 지표: v1 목표 동시 세션 ≥ 8 (설계상 상한 아님), 목록/탭이 세션 수 증가에 선형 열화 이내
- 관련: FR-017

#### [NFR-013] 터미널 렌더 확장 무중단 — Should
- 지표: 렌더 기능 확장이 엔진 경계(`ITerminalSession`) 격리 덕에 상위 오케스트레이션 재작업 없이 가능
- 관련: FR-004·FR-005 · NFR-018

### 3-5. 사용성 (Usability)

#### [NFR-014] 단일 파워유저 워크플로 효율 — Should
- 지표: 핵심 동작(세션 기동·커맨드 주입·채널 송신)이 ≤ 3 상호작용, 키보드 중심 조작 지원
- 관련: FR-011·FR-014·FR-025

#### [NFR-015] 터미널 UX 정합 — Should
- 지표: 기본 상호작용(포커스 타이핑·복사·스크롤백)이 Windows Terminal/VS Code 터미널 수준 관성 유지
- 관련: FR-009·FR-010·FR-007

### 3-6. 유지보수성 (Maintainability)

#### [NFR-016] Feature × Layer 구조 준수 — Must
- 지표: 신규 기능이 Shell/Features/Shared 3분할 + 기능 내부 Models/Views/ViewModels/Services 레이어 준수, Feature 간 직접 참조 0
- 관련: FR-040 · 아키텍처 가이드

#### [NFR-017] IPC 재구현 금지 (계약 준수) — Must
- 지표: 앱이 IPC 전송/큐/watcher 로직을 재구현하지 않고 skill_ipc_control 스크립트/파일 계약만 소비(재구현 0)
- 관련: FR-024·FR-025 · 결정 로그 #3

#### [NFR-018] 터미널 엔진 교체 가능성 — Should
- 지표: 엔진 경계 인터페이스(`ITerminalSession`) 1곳으로 격리되어 엔진↔폴백 교체 시 상위 소비자 코드 변경 최소
- 관련: FR-003·FR-004

### 3-7. 호환성 (Compatibility)

#### [NFR-019] 플랫폼 호환 — Must
- 지표: Windows(11 기준)·.NET 10·WPF·ConPTY(`CreatePseudoConsole`) 환경에서 동작
