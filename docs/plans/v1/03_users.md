# 03. 사용자 유형·페르소나 (User Types / Personas)

> 담당: plan_user_classifier · 깊이: deep · 총 UT 5(루트 1 + 운영모드 4) / P 1
> 본 문서는 RBAC 권한 조합으로 사용자 유형을 도출한다. 단, 본 제품은 단일 파워유저·로컬·인증 없음(제약 C7)이라 권한 경계가 1인으로 수렴하므로, "동일 파워유저의 운영 모드(exercised permission subset)"를 실질 분류축으로 삼아 UT를 전개하고 1개 실인물 페르소나로 구체화한다.

---

## 0. 개요

### 0-1. 목적·범위

본 문서는 `04_requirements.md`가 freeze한 8 공통 카테고리(보호 자원군)와 FR 동사(행위)를 근거로, Control Tower v1을 사용하는 **사용자 유형(UT)**과 대표 **페르소나(P)**를 정의한다.

- **정의하는 것**: 누가(사용자 유형·모드) / 무엇을 다루는가(자원×행위 권한) / 어떤 인물이 왜 쓰는가(페르소나).
- **핵심 전제(정직한 RBAC 결론)**: 제약 C7(로컬 전용·인증 없음·**단일 파워유저**)에 의해 앱에는 인증 경계가 없고 **물리적 사용자는 1명**이다. 따라서 "권한 조합으로 사용자 유형을 나눈다"는 원리를 그대로 적용하면 **granted permission set = superuser 단일 역할**로 붕괴한다. 억지로 다수 사용자 유형을 만들지 않는다.
- **채택한 분류축(대안)**: 동일 파워유저가 시간대·목적에 따라 갈아끼우는 **운영 모드(Operating Mode)** — 즉 *부여된* 권한이 아니라 그 국면에서 *실제로 행사하는(exercised)* 자원×행위 부분집합 + 목적/빈도/숙련 맥락. 이는 04가 명시한 5역량·로드맵 레이어와 자연스럽게 정렬된다.
- **경계(다른 문서와의 관계)**: FR/NFR·카테고리 정의=04(본 문서는 참조만) / 기능 분해(FN)=05 / 이 UT들의 행동 시나리오·Journey=06 / 화면=07. 본 문서는 "행위 주체"만 정의하고 자원·행위 ID는 재발번하지 않는다.
- **관리 대상 액터 주의**: 앱이 오케스트레이션하는 다수의 **Claude Code / pwsh 세션은 "비인간 액터"**로서 앱의 *제어 대상*이지 GUI *사용자*가 아니다(§4·§5에서 구분).

### 0-2. 사용자 유형 체계 (RBAC → 운영 모드)

```
[물리적 사용자 1인] -- 인증경계 없음(C7) -- granted perms = ALL(8 카테고리 full)
        |
        +- UT-001  단일 파워유저(루트/슈퍼유저)  <- RBAC상 유일한 principal
               |  (동일인이 국면별로 갈아끼우는 운영 모드 = 실질 분류축)
               +- UT-002  세션 함대 오케스트레이터   (fleet macro)
               +- UT-003  개별 세션 운영자           (terminal micro)
               +- UT-004  프롬프트·프로파일 자산 관리자 (authoring-time)
               +- UT-005  관측자                     (read-only watch)
```

- **UT-001**은 유일한 RBAC 역할(그 사람 자체, 전 권한 보유). **UT-002~005는 UT-001의 운영 모드 facet**이며 모두 `⊆ UT-001`(부여 권한은 동일, 행사 부분집합만 다름).
- 각 모드의 `권한 집합`은 **granted가 아니라 exercised**(그 국면에서 주로 행사하는 {자원: 행위})로 표기한다.
- **UT-###·P-###만 신규 발번**한다. 자원군(TRM·SES·PRF·IPC·OBS·AST·SEC·SYS)과 FR은 04가 freeze한 것을 **참조만** 한다(재번호 금지).

### 0-3. 분류축·표기 규칙

- **행위 어휘(actions)**: `spawn`(기동/생성) · `kill`(종료) · `restart`(재시작) · `inject`(커맨드/트리거 주입) · `read`(조회) · `watch`(실시간 감시) · `send`(IPC 송신) · `edit/CRUD`(생성·읽기·쓰기·삭제·이름변경) · `config`(설정) · `approve`(위험 가드 승인) · `type/copy`(터미널 입력·복사).
- **행사 강도(자원×행위 매트릭스, §2-0)**: `●` full 제어/CRUD · `◐` 부분(조회+일부 행위) · `○` read-only · `—` 미행사.
- **분류축(각 UT)**: 역할(Mode) / 목적(Goal) / 빈도(일상·간헐·상시백그라운드) / 숙련도(초·중·고) / 비중(활동 시간 점유 추정).
- **비중**: 단일 사용자이므로 "사용자 수 비율"이 아니라 **한 사람의 활동 시간이 각 모드에 쓰이는 추정 비율**이다. 모드 4종 합 ≈ 100%. UT-001은 이 합의 상위 집합이므로 비중=100%(전체).
- **포함관계(⊇/⊆)**: 상위가 하위의 행사 권한을 포함하면 표기.

| UT-003 | 개별 세션 운영자 | TRM·SES(단일)·OBS(단일) | type·copy·inject·kill | 일상 | 고 | ~30% | ⊆001 |

> UT-002~005는 **동일 인물의 국면(hat)**이다. RBAC 부여권한은 모두 UT-001과 동일하며, 위 "주 행사 자원"은 각 국면에서 *실제로 손을 대는* 자원×행위 부분집합이다.

### 1-2. 페르소나 한눈에

| P | 이름 | 소속 UT | 한 줄 정체성 | 핵심 니즈 |
|---|---|---|---|---|
| P-001 | 카이토르(본인, 가명) | UT-001 (전 모드 겸행) | 여러 Claude 세션 함대를 자기 도구로 관제하는 솔로 파워유저 | 흩어진 세션을 한 화면에서 소유·제어·협업·관측 |

> 물리적 사용자가 1명이므로 실인물 페르소나도 1개(P-001)만 발번한다. P-001은 UT-002~005 모드 전부를 시간대별로 겸행한다. 미래 확장(v2 팀 협업자·원격 사용자)은 **잠재 페르소나**로 §5에 참고 기재하되 발번하지 않는다.

## 2. 사용자 유형 상세

### 2-0. UT × 자원(FR 카테고리) 행사 매트릭스 — 사용자↔기능 매핑

각 모드가 04의 8 카테고리(=보호 자원군)를 어느 강도로 행사하는지. 대표 FR은 04 참조.

| 자원군(카테고리) | 대표 FR(참조) | UT-002 오케 | UT-003 운영 | UT-004 자산 | UT-005 관측 |
|---|---|:---:|:---:|:---:|:---:|
| TRM 터미널·렌더·탭 | FR-004·006·009·010 | — (간접) | ● | — | ○ |
| SES 세션 생성·제어 | FR-011·014·015·017 | ● | ◐ (단일) | — | ○ (상태) |
| PRF 세션 프로파일 | FR-019·020·022·023 | ◐ (기동) | — | ● (CRUD) | — |
| IPC 채널·협업·주입 | FR-024·025·026·027 | ● | — | — | ○ (뷰) |
| OBS 관측·토큰 | FR-030·031·032 | ○ | ○ (단일) | — | ● |
| AST 프롬프트 자산 | FR-033·034·035·036 | — | — | ● (CRUD) | — |
| SEC 보안·안전 | FR-037·038·039 | ● (승인) | ◐ | ◐ (경로) | — |

> **권한 커버리지**: 8 카테고리 모두 ≥1 모드가 능동 행사(●/◐)로 보유 → 누락 자원 0. (상세 검증 §6)

### 2-1. [UT-001] 단일 파워유저 (루트 / 슈퍼유저)

- **역할**: 앱의 유일한 인간 사용자·소유자. RBAC상 단 하나의 principal.
- **목적**: 자기 Claude 세션 함대를 로컬에서 완전 장악·관제.
- **빈도**: 상시 / **숙련도**: 고(도구 제작자 본인) / **비중**: 100%(전체).
- **권한 집합(RBAC granted = 전체)**: `{ TRM: 전부, SES: 전부, PRF: 전부, IPC: 전부, OBS: 전부, AST: 전부, SEC: 전부, SYS: 전부 }`.
- **포함관계**: `⊇ UT-002·UT-003·UT-004·UT-005` — 아래 4개는 이 유저의 운영 모드 facet이다.
- **비고**: 인증 경계가 없어(C7·FR-038·NFR-006) 권한 상승·격리 개념이 없다. 이하 모드 구분은 *권한 차단*이 아니라 *행동·화면 맥락*의 분리다.

### 2-2. [UT-002] 세션 함대 오케스트레이터 (Fleet Orchestrator, mode)

- **역할 / 목적**: 함대 매크로 지휘 — 여러 세션을 프로파일로 기동하고, IPC 채널로 묶어 협업시키며, 트리거 프롬프트로 스킬을 주입하고, 위험 주입을 승인한다. 04 1차 정체성("AI 세션 오케스트레이션")의 주 실행 국면.
- **빈도**: 일상(핵심 경로) / **숙련도**: 고 / **비중**: ~35%.
- **권한 집합(exercised)**: `{ SES: spawn·kill·restart·inject·목록/전환, IPC: watch·send·멤버십·스킬주입·대화뷰, PRF: 기동(read/run)·프리셋, OBS: read(함대 토큰 개관), SEC: approve(위험 가드), SYS: 셸 영역 조합 }`.
- **포함관계**: `⊆ UT-001`. 행사 관점에서 `⊇ UT-005`(관측자가 보는 OBS·IPC·상태를 오케스트레이터도 조회).
- **대표 FR(참조)**: FR-011·014·017·024·025·026·027·037.

### 2-3. [UT-003] 개별 세션 운영자 (Session Operator, mode)

- **역할 / 목적**: 단일 임베드 터미널을 손으로 직접 조작하는 마이크로 국면 — 활성 탭에 타이핑, TUI 관찰, 텍스트 선택·복사, 스크롤백 조회, 해당 세션 커맨드 주입·종료.
- **빈도**: 일상 / **숙련도**: 고 / **비중**: ~30%.
- **권한 집합(exercised)**: `{ TRM: read(렌더)·type(입력 포커스)·copy·리사이즈·스크롤백·탭전환, SES: inject(단일)·상태 read·kill, OBS: read(해당 세션 토큰) }`.
- **포함관계**: `⊆ UT-001`.
- **대표 FR(참조)**: FR-004·006·009·010·014·015·032.
- **비고**: claude는 alt-screen TUI이며 공식 WT 렌더러가 기본 렌더하므로 관찰·주입·제어가 모두 가용(제약 C10).

### 2-4. [UT-004] 프롬프트·프로파일 자산 관리자 (Asset Curator, authoring-time mode)

- **역할 / 목적**: 실행-시간이 아닌 **셋업/정비-시간** 국면 — 재사용 세션 프로파일을 짓고(CRUD), `~/.claude/{rules,skills,agents}` 프롬프트 자산을 편집하며, 앱 경로·기본값을 설정한다.
- **빈도**: 간헐(셋업·정비 시 집중) / **숙련도**: 고 / **비중**: ~15%.
- **권한 집합(exercised)**: `{ PRF: create·edit·복제·delete·as정의, AST: 트리 read·write·create/delete/rename, SEC: 자산 경로 안전(~/.claude 경계 준수), SYS: config(경로·기본 프로파일·렌더 단계) }`.
- **포함관계**: `⊆ UT-001`.
- **대표 FR(참조)**: FR-019·020·021·023·033·034·035·036·042.
- **비고**: v1 자산 편집은 파일 읽기/쓰기까지(스키마 검증은 후속, A4·후속숙제⑤). 경로 탈출 차단(NFR-008)이 이 모드의 핵심 안전 제약.

### 2-5. [UT-005] 관측자 (Observer, read-only mode)

- **역할 / 목적**: 세션들이 자율 실행되는 동안의 **수동 모니터링** 국면 — 토큰 사용량, 채널 대화 흐름, 세션 상태를 관망(read/watch)만 하고 변경하지 않는다.
- **빈도**: 상시(백그라운드 관망) / **숙련도**: 중~고(수동적) / **비중**: ~20%.
- **권한 집합(exercised)**: `{ OBS: 트랜스크립트 탐지·토큰 집계·표시(read), IPC: 채널 대화뷰·상태 watch(read, no send), SES: 상태 read, TRM: 화면 read(no input), SYS: 로그 read }`.
- **포함관계**: `⊆ UT-002 ⊆ UT-001`(오케스트레이터의 조회 권한 부분집합).
- **대표 FR(참조)**: FR-016·028·030·031·032 · NFR-022.
- **비고**: 별도 principal이 아님에도 모드로 분리하는 이유 = 화면/행동 설계상 "지시하지 않고 지켜보는" 저개입 국면이 06 Journey·07 대시보드 화면 설계에 유의미하기 때문.

## 3. 페르소나

### 3-1. [P-001] 카이토르(본인, 가명) — 단일 파워유저 / UT-001 (전 모드 겸행)

- **나이/성별 · 직업/배경**: 30대 · 솔로 개발자 겸 프롬프트 엔지니어. 여러 Claude Code 세션을 동시에 돌려 다중 프로젝트(예: `control` / `sangmin+noter` / `Proj_N`)를 병행한다. Windows 11 위에서 WPF/.NET 10으로 **자기 워크플로 도구를 직접 제작**해 쓰는 유형(dogfooding).
- **핵심 니즈**: (1) 창마다 흩어진 세션을 한 앱에서 **앱-소유로 완전 제어**, (2) 세션 간 **IPC 협업**(채널·트리거 주입), (3) 토큰·프롬프트 자산까지 **한 곳에서 관제**, (4) 의도별 **프로파일 프리셋**으로 즉시 함대 구성.
- **Pain Point**:
  - 세션 sprawl — 터미널 창이 난립해 어느 세션이 무엇을 하는지 추적 불가.
  - 외부 프로세스는 ConPTY attach가 불가(C1)해 기존 "전역 감지" 접근이 제어와 단절.
  - 토큰 소진이 불투명하고, 프롬프트 규칙 수정마다 별도 에디터로 이탈.
  - 세션 협업을 붙일 때 IPC 스킬을 매번 수동으로 발동시켜야 함.
- **기대 효과(Gain)**: 프로파일 1클릭 기동 → 탭 전환 → 커맨드/트리거 주입 → 채널 관측 → 토큰·자산 통합 관리까지 **키보드 중심 3-상호작용 이내**(NFR-014)로 완결.
- **사용 시나리오(하루)**: ①[UT-004] 아침에 프로파일 프리셋 정비 → ②[UT-002] 프리셋으로 3세션 기동, IPC 채널로 묶고 협업 트리거 주입 → ③[UT-003] 특정 세션 TUI에 직접 타이핑·복사하며 세밀 조작 → ④[UT-005] 백그라운드로 토큰·채널 대화 관망 → 필요 시 ②로 복귀.
- **기술 환경**: Windows 11 · .NET 10 · WPF · ConPTY(`CreatePseudoConsole`) · ClickOnce 배포 · `~/.claude` 자산 트리 · `skill_ipc_control` 파일 기반 채널(`inbox.log`/`.relay_url`/`.cursor_<as>`).
- **(deep) 행동 동기**: 오케스트레이션 오버헤드를 최소화하면서 다수 AI 에이전트를 지휘하는 **"관제탑" 장악감**; 자기 도구로 자기 워크플로를 완결하려는 크래프트 지향. 1차 정체성은 "터미널"이 아니라 "AI 세션 오케스트레이션"(04 결정로그 #1)임을 스스로 강하게 인지.
- **(deep) 이탈 요인(Churn Risk)**:
  - **렌더 품질**: ConPTY 셀 렌더가 Windows Terminal/VS Code 터미널보다 버벅이면(NFR-001 미달) 그냥 그것들을 쓰고 앱을 버린다(제작자=사용자라 관용이 낮음).
  - **오버헤드 역전**: "프로파일·채널 세팅 수고 > 수동 다중 창 관리"가 되면 이탈.

## 4. 역할 간 상호작용

단일 인간이므로 UT 간 상호작용은 **① 동일 인물의 모드 전환(hat-switching)**과 **② 인간→비인간 액터(AI 세션) 제어 + 세션↔세션 IPC 협업** 두 층으로 나뉜다.

### 4-1. 모드 전환 루프 (동일 인물의 국면 이동)

```
        [UT-004] ---publish profiles/assets---> [UT-002]
      (authoring)                               (orchestrate)
           ^                                         |
           |                                    spawn/inject/send
       refine assets                                 v
           |                                    [UT-003]
       [UT-005] <----watch tokens/channels----- (operate)
      (observe)  <-----------------------------------'
```

캡션: UT-004(자산 관리자)가 프로파일·프롬프트 자산을 정비해 UT-002(오케스트레이터)에게 "재료"를 넘기고, UT-002가 세션을 기동·주입·협업시키면 UT-003(세션 운영자)이 개별 세션을 직접 조작하며, 그 결과를 UT-005(관측자)가 관망한다. 이 넷은 **권한 위임이 아니라 동일 인물의 시간대별 국면 전환**이다(부여권한 동일).

### 4-2. 인간 ↔ AI 세션 액터 ↔ IPC 협업 (진짜 다-액터 구조)

```
                     [HUMAN-PU]  (UT-001)
                          |
             spawn / inject / observe  (GUI, 단일 조작자)
                          v
      +=============  Control Tower app  =============+
      |   [SESS-A] <---- IPC channel ----> [SESS-B]   |
      |      \                                /       |
      |       '--- transcript/token ----> OBS        |
      |   [SESS-C] ---- (앱-소유 ConPTY only) ----    |
      +===============================================+
              X  [EXT-PROC]  (앱 미소유 → 제어 배제)
```

캡션: 인간 파워유저 1명이 앱을 통해 N개의 **AI 세션(SESS-A/B/C = 비인간 액터)**을 기동·주입·관측하고, 세션들은 IPC 채널로 서로 협업한다. 세션은 앱-소유 ConPTY만 대상이며(제약 C1·FR-039), 앱이 소유하지 않은 외부 프로세스(EXT-PROC)는 제어에서 배제된다.

### 4-3. 검수·위임·SoD(직무 분리) 관계

- **권한 위임(상위→하위)**: 없음(N/A). UT-002~005는 UT-001의 facet이므로 위임이 아니라 자기-국면 전환.
- **검수 관계**: 인간↔인간 검수는 없다. 대신 **인간이 AI 세션 산출을 검수**한다(UT-003/UT-005가 세션 출력·토큰을 확인). 세션↔세션 협업 결과의 최종 판단자는 항상 단일 인간.
- **SoD(직무 분리) — 붕괴 주의**: 다인 RBAC라면 "위험 커맨드 주입자"와 "승인자"를 분리하겠지만(FR-037·NFR-007), 여기서는 **주입자 = 승인자 = 동일인**이라 SoD가 성립하지 않는다. 따라서 위험 가드(FR-037)는 2인 승인 통제가 아니라 **동일인의 오조작을 막는 자기-체크포인트(UI 확인 게이트)**로 설계되어야 한다 — 다중 세션에 잘못 주입하는 사고 방지가 유일한 통제선.
- **권한 충돌**: 부여권한이 1인으로 통일되어 권한 충돌(access conflict)은 발생하지 않는다. 유일한 "충돌"은 동일인의 **주의 분산**(오케스트레이터로 지시 중 엉뚱한 활성 탭에 주입) — 07 화면 설계에서 활성 세션 명시·주입 대상 확인으로 완화.

## 5. 비사용자 (Out of Scope)

| 대상 | 성격 | 제외 사유 | 향후 |
|---|---|---|---|
| **Claude Code / pwsh 세션** | 비인간 액터(관리 대상) | 앱의 *제어·관측 대상*이지 GUI를 조작하는 *사용자*가 아님. IPC로 자율 협업하나 앱 UI를 쓰지 않음 | v1 핵심 관리 대상(사용자 아님) |
| **외부 프로세스(앱 미소유)** | 배제 대상 | ConPTY attach 불가(C1·FR-039), 전역 감지 폐기(브리프 §6). 목록·주입 대상에서 제외 | 유지(배제) |
| **팀 협업자 / 동료 개발자** | 잠재 페르소나 | v1은 단일 파워유저·인증 없음(C7). 다중 사용자·권한 경계 없음 | **v2** 후보 — 도입 시 진짜 RBAC(역할·인증) 필요 |
| **원격 사용자(다른 디바이스)** | 잠재 페르소나 | 멀티디바이스 원격제어 = v2(결정로그 #2·FR-038·NFR-006). v1은 외부 포트 미개방 | **v2**(HTTP relay) |
| **세션 산출물의 최종 소비자** | 앱 밖 | 세션이 만든 코드/문서의 사용자는 앱 관제 범위 밖 | 범위 밖 |

> **잠재 페르소나(발번 보류)**: "팀 협업자"는 v2에서 인증·역할 분리가 도입될 때 비로소 별도 UT/P로 발번한다. v1에서는 P-001 단일 페르소나만 유효.
