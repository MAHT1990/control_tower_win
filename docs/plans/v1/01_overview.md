# 01. 제품 개요 (Product Overview)

> 담당: orchestrator(합성) · 깊이: deep · v1 기획 종합
> 본 문서는 14문서 기획 산출물의 최상위 요약이다. 제품 정체성·핵심 컨셉·범위·가정·성공 기준을 한 장으로 압축한다.

---

## 한 줄 정의

**Control Tower** — 로컬에서 여러 Claude Code/PowerShell 세션을 **앱이 직접 소유(ConPTY〔Windows Pseudo Console〕 임베드)** 하여, 의도별 **세션 프로파일**로 띄우고 · **IPC 채널**로 묶어 협업시키며 · **토큰·프롬프트 자산**까지 한 곳에서 관제하는 **단일 파워유저용 AI 세션 관제탑** (WPF/.NET 10 데스크톱).

---

## 제품 정체성

- **1차 정체성 = AI 세션 오케스트레이션 콘솔.** 터미널 임베드(ConPTY)는 그 가치를 실현하는 **substrate(토대)** 이지 목적이 아니다(결정 로그 #1).
- **차별축(경쟁 부재)**: 조사한 오케스트레이터(Claude Squad·Conductor·Vibe Kanban)는 전부 git worktree "격리" 모델 — 세션이 서로 대화하지 않는다. Control Tower는 `skill_ipc_control` 재사용으로 **세션 간 IPC 채널 협업**을 제공하는 유일한 도구다(02 D1). 여기에 **Windows 네이티브 데스크톱**(경쟁은 macOS/웹/TUI) + **토큰 관측·프롬프트 자산 편집 내장**이 결합된 포지션은 시장에서 비어 있다.
- **타깃**: 여러 Claude 세션 함대를 병행하는 **단일 파워유저(본인, dogfooding)**. 로컬 전용·인증 없음.

---

## 핵심 컨셉 (5축)

> **주의 — "5축" ≠ "5역량"**: 아래 핵심 컨셉 5축(①~⑤)은 `00_meeting_brief`·04가 쓰는 "5역량"(그쪽 ⑤=멀티디바이스=v2 범위)과 번호만 겹칠 뿐 **서로 다른 ①~⑤ 목록**이다 — 같은 번호끼리 대응하지 않는다.

| 축 | 내용 | 근거 문서 |
|---|---|---|
| **① 앱-소유 임베드 터미널** | 앱이 pwsh를 직접 spawn·소유하여 입력(주입)·출력(캡처)을 장악. 외부 프로세스 attach 불가 → 앱-소유 세션만 대상. 터미널 렌더링은 공식 Windows Terminal 렌더러(EasyWindowsTerminalControl)에 위임. | 10 · 07 |
| **② 세션 프로파일 애그리거트** | 의도태그·초기명령·작업디렉토리·주입스킬·채널멤버십·claude자동실행 + `as`(IPC 식별자). 저장·재사용·프리셋 1클릭 함대 기동. 오케스트레이션의 중심 도메인. | 05 · 09 |
| **③ IPC 오케스트레이션** | `skill_ipc_control`의 GUI 프론트엔드 — 채널 파일 watch/read + `send.cmd` 재사용. IPC 미보유 세션에 **트리거 프롬프트를 입력 파이프로 전송**해 스킬을 주입(재구현 금지). | 05 · 09 |
| **④ 관측** | 세션 jsonl(JSON Lines) 트랜스크립트를 증분 파싱해 토큰 사용량 집계·표시. | 09 · 10 |
| **⑤ 프롬프트 자산 편집** | `~/.claude/{rules,skills,agents}` 파일 트리 브라우저 + 에디터(경로 탈출 가드, 원자적 저장). | 07 · 09 |

**의존 함의**: 입력 경로(주입)는 렌더 완성도와 무관하게 확보되므로 **커맨드/스킬 주입은 화면 관찰에 볼모잡히지 않는다** → 오케스트레이션(L2)이 터미널 렌더에 독립적으로 진행된다.

## 범위 (Scope)

**v1 In-Scope**: 로컬 전체 — ConPTY 임베드·세션 소유/제어·세션 프로파일·IPC 오케스트레이션(로컬 파일 기반)·토큰 관측·프롬프트 자산 편집.

**Out-of-Scope (v1 제외)**:
- **멀티디바이스 원격제어**(HTTP relay 기반 디바이스간 IPC) = **v2**. v1은 수신 포트 0(외부 미노출).
- 프롬프트 자산의 **스키마 검증**(v1은 파일 읽기/쓰기까지).
- **pane 분할**(FR-008)·**세션 상태 완전 복원**(FR-043) — 후속 마일스톤/Could.
- 외부 터미널 새창 실행 — v1 제외(보조 잔류는 검토).

## 성공 기준

| 관점 | 성공 기준 |
|---|---|
| **정체성** | 수동 다중 터미널 창 관리 → 앱-소유 관제탑으로 전면 이전(하루 오케/운영/관측이 앱 내 완결) |
| **Aha Moment** | 프리셋 1클릭 함대 기동 + 트리거 주입으로 세션 협업 자동 개시(≤3 상호작용) |
| **차별화 검증** | 세션 간 IPC 협업·토큰 관측·자산 편집 — 경쟁 부재 3축 구현 |
| **품질 게이트** | 크래시 격리 100%(NFR-009) · 렌더 실시간 무버벅임(NFR-001) · 주입 ≤100ms(NFR-002) · 경로 탈출 0·포트 0 |
| **임계경로** | Phase 0 압축 PoC(엔진 통합·native 배포·주입/캡처·세션 수명) 통과 = 터미널 substrate 확정 |
| **MVP 출시선** | L0(터미널 substrate) + L1(세션 소유·제어) 완성(Gate2) → ClickOnce v1 MVP |

**이탈 리스크(P-001 Churn)**: 터미널 이질감 → 기성 터미널로 회귀 / 세팅 오버헤드 역전 / 관측 부정확 → 신뢰 붕괴. → 공식 WT 렌더러 품질·프리셋 즉시성·관측 정확성이 사수 지점.
